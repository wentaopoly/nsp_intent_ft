NSP Intent 智能配置系统

Nokia NSP · 自然语言 → API-Ready JSON · 9 种 Intent 类型
2026-04

问题：手动配置 NSP Intent 的痛点

  • 每个 Intent JSON 包含 数十到上百 个嵌套字段
  • YANG schema 定义严格：类型、范围、枚举、正则模式、max-elements
  • 跨字段语义约束（SDP 双向绑定、VLAN 匹配、设备 IP 不重复）不在 YANG 中表达
  • 9 种服务类型 × 多站点组合 → 手动编写极易出错
  • NSP API 校验在服务器端 (JS-scripted mapping-engine)，反馈滞后

目标：用自然语言描述网络服务需求，自动生成合规 JSON

方案概览：NL → JSON 自动化

M2.5: <think> Chat Template 修复

Qwen3 模型在推理时插入 <think>...</think> 思考标记

修复前 (Token 序列):

  [system] [user]  <think> 推理内容... </think>  {"intent_type": ...}
                   ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
                   模型输出包含 think block
                   JSON 提取失败或被截断

  训练时 chat template 对齐 → assistant turn 从 { 开始
  推理时 enable_thinking=False + 正确 stopping criteria

根因分析：随机字段不在指令中

核心问题：生成器内部随机生成的字段值没有出现在自然语言指令中

# 旧代码 (有 bug):
def generate_evpn_epipe_values():
    values["evi"] = random_evi()            # 随机生成
    values["evpn-type"] = random.choice(["mpls", "vxlan"])  # 随机
    # ... 但指令模板没有 {evi} 和 {evpn_type} 占位符!

# 模型看到的指令:
"Create EVPN-Epipe service 'svc-001' for customer 10..."
# 模型需要预测的 fill_values 包含 evi=5823 -- 但指令里没提到!
# 模型只能猜 → val_acc 下降

影响：模型无法从指令推导出这些"隐藏"字段，被迫猜测

修复：纯函数生成器 + 确定性推导

# 修复后: roll args 一次，传给生成器和模板
def _roll_evpn_epipe_args():
    """Roll all instruction-visible args once."""
    return {
        "service_name": ..., "customer_id": ...,
        "evi": random_evi(),           # 出现在指令中
        "evpn_type": random.choice(["mpls", "vxlan"]),  # 出现在指令中
        ...
    }

args = _roll_evpn_epipe_args()
values = generate_evpn_epipe_values(**args)   # 确定性转换
instruction = template.format(**args)          # 同一组参数

设计原则：

  • 所有出现在 fill_values 中的值必须可从指令推导
  • 生成器是纯函数：相同输入 → 相同输出
  • 随机性只在 _roll_*_args() 中发生一次
  • SDP ID 等派生值通过 derive_sdp_id() 确定性计算

问题：E-Tree val_acc 79%

E-Tree 的特殊挑战：

  • Root-Leaf 拓扑：1 个 root site + 2-3 个 leaf sites
  • SDP 不是全网状 (full mesh)，而是 root ↔ 每个 leaf 的星型连接
  • 旧实现使用了 VPLS 风格的 full mesh SDP → 语义错误
# Full mesh (错误 -- VPLS 风格):
SDP: root↔leaf0, root↔leaf1, leaf0↔leaf1  (叶子间不应有 SDP)

# Star topology (正确 -- E-Tree 风格):
SDP: root↔leaf0, root↔leaf1  (仅 root 与每个 leaf 之间)

另一个问题：指令中 leaf 站点的描述格式模糊

# 模糊: "leaf sites on devices 10.0.0.1, 10.0.0.2 with ports ..."
# 不清楚哪个 device 对应哪个 port

# 修复后: "device 10.0.0.1 on port 1/2/c3/1, device 10.0.0.2 on port 1/2/c4/1"

根因：SDP 全网状 + 格式歧义

问题根因影响
SDP 拓扑错误 照搬 VPLS full-mesh SDP 生成逻辑 多余 SDP 条目，Tier 4 语义失败
设备-端口对应模糊 指令模板用列表式罗列 devices 和 ports 模型无法确定哪个 device 用哪个 port
VLAN 分配不一致 root 和 leaf 的 inner/outer VLAN 生成独立 val_acc 因 VLAN 值不匹配下降

修复：Root-Leaf SDP + 明确格式

  • SDP 拓扑：改为 root ↔ leaf 星型，每对生成双向 SDP
  • 指令格式：采用 "device X on port Y" 的一对一描述
  • VLAN：统一生成一组 outer/inner VLAN，root 和所有 leaf 共享
# 修复后的指令模板:
format_etree_leaves_desc(leaf_devices, leaf_ports):
  # "device 10.0.0.5 on port 1/2/c3/1, device 10.0.0.8 on port 1/2/c4/1"
  # 每个 device-port 对明确绑定，消除歧义

结果：etree val_acc 从 79% 提升至 100%

总结

核心成果

  • 基于 Qwen3.5-9B + LoRA 实现自然语言到 NSP intent JSON 的自动生成
  • 覆盖 9 种 intent 类型，2,640 训练样本，330 测试样本
  • 建立 YANG 驱动的 4 层验证体系，离线保证结构合规性
  • 最终测试结果：330/330 test + 11/11 golden = 100% 全指标

关键经验

  • Chat template 对齐是 LLM 微调的隐形杀手 -- <think> token 偏移可致全面失败
  • 纯函数生成器原则：训练数据中每个目标值必须可从输入推导
  • 拓扑感知：不同服务类型的 SDP 连接模式不同 (full mesh vs. star)
  • YANG schema 是 "single source of truth"，但领域知识 (Sarvesh) 不可或缺
